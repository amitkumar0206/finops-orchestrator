| F-48 | Missing Tenant Isolation on Analytics Cost Explorer Calls (HIGH-15) | HIGH | **The bug:** all four `get_cost_and_usage()` call sites in `backend/api/analytics.py` had no `Filter` param. Cost Explorer, called without a `LINKED_ACCOUNT` filter from a management-account role, returns spend across **every** linked account. Any authenticated user — regardless of org — saw every tenant's AWS costs. `check_historical_data_availability` leaked the cross-tenant total directly in the response body (`f"...Total cost: ${total_cost:,.2f}"`). The background task `_load_historical_data_to_cache` ran fully unscoped (signature was `(months: int)` — no context) and wrote results under tenant-agnostic Valkey keys. **Attribute reality:** the finding said `context.account_ids`; the actual attribute is `context.allowed_account_ids: List[str]` at `request_context.py:63` — 12-digit AWS account IDs populated by `AccountScopingMiddleware` from `aws_accounts WHERE tenant_org_id = :org_id AND status = 'ACTIVE'`, intersected with `account_permissions` for non-admins, further intersected with the active saved-view selection. An empty list is a LEGITIMATE state (no org, no permissions, empty saved-view intersection) and must fail-closed — the unscoped query is exactly the bug. **Fix — two module-level helpers + 4 call sites + signature + cache keys:** (1) **`_build_account_filter(allowed_account_ids) -> dict`** — raises `HTTPException(403, "No AWS accounts in scope for this request")` on empty list, otherwise returns `{'Dimensions': {'Key': 'LINKED_ACCOUNT', 'Values': list(allowed_account_ids)}}`. Defensive `list(...)` copy so mutating `Values` doesn't mutate the `RequestContext` instance. Single helper = single policy point; adding an admin-bypass later is one edit. (2) **`_scope_cache_key(allowed_account_ids) -> str`** — `hashlib.sha256(",".join(sorted(ids)).encode()).hexdigest()[:16]`. Sorted so the same account-set in different middleware load-order produces the same key. Hashed so `valkey-cli KEYS *` doesn't leak which 12-digit account IDs a tenant owns. 16 hex chars = 64 bits, negligible collision. (3) **All three handlers call `_build_account_filter(context.allowed_account_ids)` as the FIRST LINE of the body** — critically, BEFORE the `try:` blocks. `check_historical_data_availability` has `except Exception → 500`; `get_data_sources_info` has `except Exception → return {"error": ...}` (a 200 with an error dict, not a raise). `HTTPException` IS an `Exception`. A 403 raised inside those blocks would be swallowed as 500 or as a 200-with-error-body respectively — wrong status, and the user learns nothing about WHY. Check-before-try means the 403 propagates directly to FastAPI. (4) **Four `Filter=account_filter` additions** — L77 (historical-availability), L132 + L144 (background task monthly + daily), L264 (data-sources ping). Zero precedent existed: `LINKED_ACCOUNT` had **zero** matches in `backend/` before this fix. (5) **Background-task signature change:** `(months: int)` → `(months: int, account_filter: dict, scope_key: str)` — both REQUIRED, no defaults. The task runs via `BackgroundTasks.add_task` AFTER the response is sent; `RequestContext` is gone. The handler builds + validates in-request (so empty-scope 403s BEFORE `add_task` is ever called — the task is never scheduled unscoped), then passes both at schedule time: `add_task(_load_historical_data_to_cache, months, account_filter, scope_key)`. (6) **Valkey key tenant segment** — `f"analytics:monthly:{start}:{end}"` → `f"analytics:monthly:{scope_key}:{start}:{end}"` (same for `daily`). The latent cross-tenant bug the fix would otherwise EXPOSE: scope the CE call but not the cache key → tenant A's filtered data writes under a date-only key, tenant B's next write overwrites it. No read path exists in this file TODAY (write-only `set()`), but fixing write-side correctness now is cheaper than unwinding cross-tenant cache state later. Scope segment goes immediately after the namespace prefix (not appended) so prefix scans like `analytics:monthly:*` stay naturally partitioned. **Scope check ordering vs. other validation:** `initialize_historical_cache` validates `months` ∈ [1,13] → 400. The scope check runs FIRST. A zero-scope user with `months=0` gets 403, not 400 — they don't learn that their months param is wrong; they don't learn anything. Security precondition before business validation. **Out of scope, noted:** `health.py:399` also has an unscoped CE call — but health-check is unauthenticated by design, no `RequestContext` exists to scope with. Left untouched. **Collateral — 2 test files:** `tests/unit/api/test_analytics_security.py` — `context_authenticated` fixture gained `allowed_account_ids=["111111111111", "222222222222"]` (two accounts so tests can assert the FULL list reaches CE, not just the first). All 15 existing tests stayed green post-fixture. `tests/unit/api/test_exception_sanitisation.py` — 4 inline `RequestContext(...)` constructions in `TestAnalyticsExceptionSanitisation` (testing HIGH-3, a different feature) gained `allowed_account_ids=["111111111111"]` — without it they 403 before reaching the exception handlers they're actually testing. **Tests — 21 new across 5 classes.** `TestAccountFilterHelpers` (8): `test_build_account_filter_returns_linked_account_dimension` (exact dict-shape equality — CE requires `Dimensions.Key == 'LINKED_ACCOUNT'`, not `'LINKED_ACCOUNT_NAME'`, not `Tags`), `test_build_account_filter_raises_403_on_empty_list` (**primary fail-closed** — 403 not ClientError), `test_build_account_filter_preserves_all_accounts` (no truncation), `test_build_account_filter_copies_input_list` (defensive copy — mutating `Values` doesn't mutate `RequestContext`), `test_scope_cache_key_is_stable_for_same_set` (sorted-before-hashed — same set, different order, same key), `test_scope_cache_key_differs_across_tenants`, `test_scope_cache_key_does_not_leak_account_ids` (raw 12-digit ID absent from key), `test_scope_cache_key_is_hex_and_bounded` (16 hex, no `:` separators to clash with key format). `TestHistoricalAvailabilityScoping` (3): `test_ce_call_filtered_by_account_scope` (**PRIMARY HIGH-15 REGRESSION** — `call_args.kwargs["Filter"]` exact-equals the expected dict, not just `"Filter" in kwargs`), `test_empty_scope_returns_403_before_ce_call` (`create_aws_client.assert_not_called()` — scope check is in the right place), `test_403_not_downgraded_to_500` (pins check outside the try). `TestInitializeCacheScoping` (3): `test_background_task_receives_account_filter` (**PRIMARY background-path regression** — `add_task.call_args.args` positionally: `(func, 12, filter_dict, scope_key)`; The subquery short-circuits to zero rows for non-owners; the thread simply doesn't exist from their perspective. **Silent filter** — no exception, no audit event (that's the API layer's job). Parameter position: 2nd, immediately after `thread_id`. `DEVELOPER_GUIDE.md:1705` example snippet updated. **Tests** — 25 new in `tests/unit/services/test_conversation_manager_isolation.py`, four groups. `TestConversationManagerSignatures` (9): `test_method_requires_user_id[*7]` — `inspect.signature()`, `"user_id" in params` AND `param.default is inspect.Parameter.empty` (required, not optional); Implemented formula: `hops[-trusted_proxy_count]`. **Fix — new shared helper `backend/utils/client_ip.py:get_client_ip()` (single source of truth):** trusted-proxy-depth semantics — split XFF on comma, strip whitespace, filter empty segments (malformed `,,1.2.3.4,` observed in the wild), take `hops[-N]` where N = `settings.trusted_proxy_count`. **Edge cases:** `N=0` → XFF entirely untrusted (no proxy to append a verified entry), ignore header, use TCP peer; XFF header absent → TCP peer (proxy bypass or local dev); `len(hops) < N` → partial chain (proxy bypassed or N misconfigured), don't trust, TCP peer; `request.client is None` → literal `"unknown"` (test mocks, Unix sockets — crashing SHA-256 key-derivation is worse). Explicit `trusted_proxy_count` arg overrides settings for test determinism. **Both call sites delegate:** `login_throttle.py:110-115` → `return get_client_ip(http_request)` (3-line body, security comment pointing to the fix); `rate_limiting.py:54-63` → `client_ip = get_client_ip(request)` (rest of `_get_client_key` — auth-user extraction, key assembly — unchanged). **Per-client SaaS configurability:** `settings.py:253` adds `trusted_proxy_count: int = Field(default=1, ge=0, le=5, env="TRUSTED_PROXY_COUNT")` with topology docs (0 = direct/local-dev, 1 = single ALB, 2 = CloudFront→ALB, 3 = CloudFront→ALB→gateway). **Infrastructure — both deployment paths surfaced** (F-39 established the dual-file requirement): `ecs-services.yaml` adds `TrustedProxyCount` CloudFormation Parameter (`Type: Number, Default: 1, MinValue: 0, MaxValue: 5`) + `Environment` entry `TRUSTED_PROXY_COUNT: !Ref TrustedProxyCount` — tenants with CloudFront+ALB pass `TrustedProxyCount=2` at stack deploy; `task-def.json` adds `{"name": "TRUSTED_PROXY_COUNT", "value": "1"}` matching the current single-ALB topology. **Existing test inverted:** `test_rate_limiting.py:86-95` (`test_uses_forwarded_ip`) was **pinning the vulnerable behavior** — asserting leftmost XFF entry used — the same pattern as F-41 where `test_blacklist_check_failure_allows_token` pinned fail-open. Renamed to `test_uses_trusted_xff_entry_not_spoofed_leftmost`, fixture rewired with 3 distinct IPs (spoofed leftmost `1.1.1.1`, ALB-appended real `203.0.113.1`, TCP peer `10.0.0.99`) so assertions unambiguously prove trusted-depth parsing. **Tests — 31 new across 3 files:** `tests/unit/utils/test_client_ip.py` (26): `TestSingleALB` (5 — single-entry, **spoofed-prefix-ignored primary regression** asserting `hops[-1]` not `hops[0]`, 10-spoof chain, whitespace strip, TCP-peer-not-returned-when-XFF-present); `TestCloudFrontPlusALB` (3 — depth=2 returns `hops[-2]`, spoofed-prefix-with-depth-two, doesn't-return-CloudFront-IP); `TestDirectConnection` (3 — **depth=0 secondary regression** asserting XFF ignored entirely, no-XFF, negative-depth defensive); `TestFallbacks` (8 — XFF absent, chain-shorter-than-depth, boundary `len==depth`, empty-segments-filtered, all-empty-segments, empty-string XFF, `client=None` → `"unknown"`, `_tcp_peer` directly); `TestSettingsIntegration` (4 — settings-driven path via `patch("backend.utils.client_ip.get_settings")` for depth 1/2/0, explicit-override-beats-settings); `TestNoLeftmostXffPatternInMiddleware` (3 — **AST tripwire**: middleware-dir-exists sanity, **walks every `middleware/*.py` for `Subscript(Call(Attribute(attr='split'), args=[Constant(',')]), Constant(0))`** with actionable failure message naming file:line → snippet, positive pin that both middlewares `ImportFrom backend.utils.client_ip` so inlined-but-correct parsing can't drift). `tests/unit/middleware/test_rate_limiting.py` (+2, 1 inverted): inverted test above + `test_spoofed_xff_does_not_evade_rate_limit` (**end-to-end attack regression** — `RateLimiter(limit=3)`, 3 requests from same real rightmost IP with 3 different spoofed leftmost prefixes → all allowed, 4th with yet-another spoof → **blocked**; pre-fix each spoof got a fresh bucket → unlimited). `tests/unit/middleware/test_login_brute_force_protection.py` (+3): `TestXffSpoofingDoesNotBypassIpLayer` — `test_spray_with_rotating_spoofed_xff_blocked` (**HIGH-35 primary regression** — real attacker IP constant in rightmost position, 20 spray attempts across 20 emails each with a different spoofed leftmost prefix → 21st blocked by Layer 1;
