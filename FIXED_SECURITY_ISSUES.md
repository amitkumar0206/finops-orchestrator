| F-42 | Missing CSRF Protection (HIGH-3) — **NOT APPLICABLE, closed by verification** | HIGH | **No code change.** The audit finding was factually correct — there is no CSRF middleware and `cors_allow_credentials` defaults to `True` (`settings.py:52`) — but the risk assessment assumed cookie-based authentication. **Verification performed:** (1) grep `set_cookie\|response.set_cookie\|httponly` across `backend/` → **zero matches**; (2) grep `cookie\|Cookie` across `backend/` → only match is the `cors_allow_credentials` Field description string in `settings.py` (documentation, not usage); (3) confirmed `backend/middleware/authentication.py:199-200` reads **only** the `Authorization` header: `auth_header = request.headers.get("Authorization")` → `extract_token_from_header(auth_header)` — no cookie extraction anywhere in the auth pipeline; (4) confirmed `backend/api/auth.py` login response returns the JWT in the **JSON body** (`access_token` field) for client-side storage, not in a `Set-Cookie` header. **Why CSRF doesn't apply:** CSRF attacks work because browsers **automatically attach cookies** to every request matching the cookie's domain, including requests triggered by `<form action="https://api.victim.com/...">` on a malicious page. The `Authorization` header is **never sent automatically** — JavaScript must set it explicitly per-request. A page at `evil.com` cannot read the JWT from `app.victim.com`'s localStorage (blocked by same-origin policy) and cannot make the browser auto-attach it. **`allow_credentials=True` is correct, not a misconfiguration** — without it, browsers drop the `Authorization` header on cross-origin requests (per the Fetch spec, "credentials" includes the `Authorization` header, not just cookies). The separate concern (wildcard origin + credentials) is already validated at `settings.py:710` which raises a CRITICAL issue if `"*" in origins and cors_allow_credentials`. **Re-open condition:** if cookie-based authentication is ever introduced (e.g., `Set-Cookie: session=...` for web clients), this finding becomes valid and requires CSRF middleware + `SameSite=Strict`. |
